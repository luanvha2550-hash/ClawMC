# Relatório de Correções — 26/06/2026

Documento com **todas as alterações necessárias** para o ClawMC voltar a funcionar.
Use este registro para montar o commit/PR da atualização no GitHub.

---

## Resumo do problema

O bot não iniciava de forma alguma. A investigação seguiu a cadeia de causas-raiz
abaixo (cada etapa revelava a próxima):

1. `npm start` falhava imediatamente com `ERR_MODULE_NOT_FOUND: Cannot find package 'zod'`.
2. `node_modules/` estava **completamente vazio** — as dependências não estavam instaladas.
3. `npm install` **falhava** antes de terminar por causa dos módulos nativos
   (`better-sqlite3`, `sqlite-vec`, `isolated-vm`).
4. Causa-raiz ambiental: a máquina está com **Node v25.6.0**, cujo V8 exige **C++20**.
   As versões pinadas dos módulos nativos (`better-sqlite3@9.6.0`, `sqlite-vec@0.1.7`,
   `isolated-vm@4.7.2`) são de início de 2024 e:
   - não publicam pré-compilados (prebuilds) para a ABI do Node 25;
   - não declaram `/std:c++20` no build, então a compilação via MSBuild falha com:
     ```
     v8config.h(13,1): error C1189: "C++20 or later required."
     ```
5. Após corrigir o ambiente, surgiram **4 bugs de código** que impediam o boot no Windows.

## Alterações de código (estas vão no commit)

### 1. `src/index.js` — guard de entry-point quebrado no Windows
**Sintoma:** após instalar as deps, o processo iniciava e **terminava em 0,
sem nenhum log**, sem criar o banco. O bot nunca chegava a rodar.

**Causa-raiz:**
```js
// ANTES — não funciona no Windows
if (import.meta.url === `file://${process.argv[1]}`) { main()... }
```
No Windows, `process.argv[1]` é `D:\...\index.js` (barras invertidas, sem barra
extra), então `file://D:\...` nunca bate com `import.meta.url` (`file:///D:/...`).
Resultado: `main()` **nunca era chamado**.

**Correção:** usar `pathToFileURL` (compatível Windows + POSIX):
```js
import { pathToFileURL } from 'node:url';
// ...
if (import.meta.url === pathToFileURL(process.argv[1]).href) { main()... }
```

### 2. `src/memory/database.js` — export `runMigrations` inexistente
**Sintoma:** `SyntaxError: The requested module './memory/database.js' does not
provide an export named 'runMigrations'`.

**Causa-raiz:** `index.js` importa `runMigrations` de `./memory/database.js`,
mas esse módulo só exportava `initDatabase`, `getDatabase`, etc. A API real de
migrations é a classe `MigrationManager` em `./memory/migrations.js`.

**Correção:** importar `MigrationManager` e expor um wrapper `runMigrations(db)`:
```js
import { MigrationManager } from './migrations.js';
// ...
export async function runMigrations(dbInstance) {
  const manager = new MigrationManager(dbInstance);
  await manager.migrate();
}
```

### 3. `src/utils/environmentCheck.js` — `require()` em módulo ESM
**Sintoma:** a checagem de ambiente reportava `better-sqlite3: ✗` e
`sqlite-vec: ✗` como **indisponíveis**, mesmo ambos estando instalados e
funcionais — o que abortava o boot.

**Causa-raiz:** o projeto é ESM (`"type": "module"`), onde `require` **não existe**.
Cada `require('better-sqlite3')` lançava `ReferenceError`, capturado pelo
`try/catch` e tratado como "módulo ausente" (falso negativo).

**Correção:** trocar `require()` por `await import()` (a função já é `async`):
```js
// ANTES
require('better-sqlite3');
// DEPOIS
await import('better-sqlite3');
```
Aplicado para `isolated-vm`, `better-sqlite3` e `sqlite-vec`.

### 4. `src/utils/timeoutManager.js` — API incompatível com `index.js`
**Sintoma:** `TypeError: this.timeoutManager.setTimeout is not a function`.

**Causa-raiz:** `index.js` usa timeouts **nomeados**:
```js
this.timeoutManager.setTimeout('bot-connect', cb, 30000);
this.timeoutManager.clearTimeout('bot-connect');
```
Mas a classe `TimeoutManager` só oferecia `createTimeout(cb, ms, op)` /
`cancel(id)` (por id, não por nome) — e esses métodos não eram usados em lugar
algum do código.

**Correção:** adicionar a API nomeada à classe:
- novo `Map` `this.namedTimeouts` no construtor;
- método `setTimeout(name, callback, ms)` — registra por nome, substituindo
  qualquer timeout anterior com o mesmo nome;
- método `clearTimeout(name)` — cancela por nome;
- `clearAll()` e `cancelAll()` atualizados para limpar também `namedTimeouts`.

### 5. `src/index.js` — `CircadianEvents.start()` / `.on()` inexistentes
**Sintoma:** com o servidor online, o bot conectava mas caía em
`TypeError: this.circadian.start is not a function` logo ao iniciar o core.

**Causa-raiz:** dois pontos no `index.js` não casavam com a classe
`CircadianEvents`:
- `this.circadian.start()` — a classe **não tem `start()`**; o registro do
  listener de `time` já é feito no construtor.
- `this.circadian?.on?.('event', ...)` — a classe **não tem `on()`**. O `?.`
  tornava isso um no-op silencioso, então os eventos circadian nunca chegavam
  ao `handleCircadian`. A classe emite os eventos no **bot** como evento
  `'circadian'` (via `bot.emit('circadian', { event, ...data })`).

**Correção:**
- removida a chamada `this.circadian.start()` (construtor já cuida do setup);
- trocado `this.circadian?.on?.('event', cb)` por
  `this.bot.on('circadian', cb)` — agora `handleCircadian` recebe de fato os
  eventos (confirmado: `[Circadian] Nightfall - initiating safety protocol`).

### 6. `src/skills/index.js` + `tests/__mocks__/mineflayer-pathfinder.js` — skills não carregavam no Windows
**Sintoma:** `Loaded 0 base skills` — todas as skills falhavam ao carregar.

**Causa-raiz (duas partes):**
1. `loadBaseSkills` fazia `await import(path.join(skillsDir, file))`. No Windows,
   `path.join` gera `src\skills\base\craft.js` (barras invertidas) —
   **especificador inválido em ESM** → `Invalid module ... is not a valid package name`.
   **Fix:** `await import(pathToFileURL(join(skillsDir, file)).href)`.
2. As 9 skills que usam pathfinder importavam `import { GoalNear } from 'mineflayer-pathfinder'`.
   `mineflayer-pathfinder` é **CommonJS** (named imports ESM não resolvem) **e**
   `GoalNear`/`GoalBlock` não são exports top-level — ficam sob `goals`.
   **Fix:** `import pathfinder from 'mineflayer-pathfinder'; const { GoalNear } = pathfinder.goals;`
   em `attack.js, collect.js, come.js, craft.js, escape.js, follow.js, mine.js, store.js, walk.js`.
3. O mock `tests/__mocks__/mineflayer-pathfinder.js` tinha default export flat
   (sem `.goals`) → `pathfinder.goals` era undefined nos testes. **Fix:** default
   export agora espelha o módulo real: `{ pathfinder, Movements, goals: { ... } }`.

**Resultado:** `Loaded 12 base skills from ./src/skills/base`.

### 7. `src/memory/embeddings.js` — modelo de embeddings carregava fp32 (425MB) em vez de quantizado
**Sintoma:** `Protobuf parsing failed` quando o cache estava corrompido; e mesmo
são, o `@huggingface/transformers` v3 ignorava `quantized: true` e carregava o
modelo **fp32 de 425MB** (log: `dtype not specified ... using fp32`).

**Causa-raiz:** na v3 do `@huggingface/transformers`, a opção `quantized` foi
substituída por `dtype`. Passar `quantized: true` virou no-op → default fp32.

**Fix:** trocar `quantized: this.quantized` por `dtype: this.quantized ? 'q8' : 'fp32'`.
Resultado: modelo carrega como `model_quantized.onnx` (~118MB) —
`Local embedding model loaded (384 dimensions)`.

> Obs: o erro `Protobuf parsing failed` relatado inicialmente era **cache
> corrompido** (download interrompido deixou `model.onnx` truncado em 142MB
> em vez dos 425MB completos). Basta apagar
> `node_modules/@huggingface/transformers/.cache/Xenova/multilingual-e5-small`
> e deixar o download terminar sem interromper. Com o fix do `dtype`, agora
> baixa o quantizado (~118MB) no lugar.
